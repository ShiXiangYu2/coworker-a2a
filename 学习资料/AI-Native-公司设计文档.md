# AI Native 公司设计文档

> 基于「地鼠精灵」创业实践整理，与 CoWorker+A2A 项目映射分析

---

## 一、核心理念：AI Native 不是用 AI 工具，而是把 AI 训练成岗位员工

### 1.1 传统模式 vs AI Native 模式

| 维度 | 传统模式 | AI Native 模式 |
|------|----------|----------------|
| AI 定位 | 效率工具 | 组织结构的一部分 |
| 使用方式 | 员工自己用 AI | 公司把 AI 训练成岗位员工 |
| 能力归属 | 能力长在员工身上 | 能力沉淀到 Skill 里 |
| 驱动方式 | 靠牛人驱动 | 靠系统驱动 |
| 可替代性 | 员工离职能力带走 | Skill 持续进化，新人快速上手 |

### 1.2 AI Native 的三层含义

```
第一层：AI 承担岗位
  └─ 不是「员工自己随便用 AI」，而是公司把 AI 训练成有岗位职责的数字员工

第二层：人从执行者变成管理者
  └─ 人负责判断、决策、沟通；AI 负责执行、整理、记录

第三层：系统驱动而非牛人驱动
  └─ 流程、系统、AI 员工、人类员工共同组成一张网，公司持续运转
```

## 二、Skill 体系：AI 员工的岗位培训手册

### 2.1 Skill 的定义

**Skill = AI 的岗位说明书**，包含以下要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| 身份定义 | 你是谁 | 「我是谈单鼠，负责销售和客户沟通」 |
| 职责范围 | 你负责什么 | 整理客户需求、生成方案、辅助跟进 |
| 工作流程 | 遇到什么情况该怎么做 | 客户线索进来后，先整理信息，再生成方案 |
| 输出标准 | 做到什么程度算合格 | 方案必须包含痛点、MVP、报价结构 |
| 行为边界 | 哪些话能说，哪些不能说 | 不能承诺交付时间，不能泄露报价底线 |
| 禁区 | 绝对不能做的事 | 不能直接成交，不能修改合同条款 |

### 2.2 Skill 创建方法

```
1. 自然语言描述岗位职责
   └─ 告诉 AI 该做什么

2. AI 自己写好 Skill
   └─ 生成完整的岗位说明书

3. 使用过程中持续改进
   └─ AI 员工自己改进自己，逐步完善
```

### 2.3 Skill 的价值

| 价值 | 说明 |
|------|------|
| 可训练 | 可以不断优化和改进 |
| 可复用 | 相同岗位可以直接复用 |
| 可交接 | 新人可以通过 Skill 快速上手 |
| 可管理 | 可以统一管理和版本控制 |

## 三、AI 员工体系：围绕业务流协作

### 3.1 核心角色

| 角色 | 职责 | 对应系统 |
|------|------|----------|
| **谈单鼠** | 销售和客户沟通 | LLM Router + Agent Routing |
| **项目鼠** | 项目交付和进度管理 | Task Executor + Multi-Agent |
| **运营鼠** | 内容生产和运营动作 | Deliverable System + Result Summarizer |
| **总控** | 统一入口和信息底座 | Orchestrator + Harmony System |

### 3.2 业务流程

```
客户线索
  ↓
[谈单鼠] ──→ 需求整理 + 方案生成 + 报价结构
  ↓
[CRM 系统] ──→ 线索记录 + 跟进状态
  ↓
客户确认合作
  ↓
[项目鼠] ──→ 任务拆解 + 排期 + 风险提醒
  ↓
[PMS 系统] ──→ 项目进度 + 负责人 + 交付节点
  ↓
项目完成
  ↓
[运营鼠] ──→ 案例包装 + 内容资产
  ↓
反哺获客 ──→ 新客户线索
```

### 3.3 人 + AI 协作模式

| 角色 | 人的职责 | AI 的职责 |
|------|----------|-----------|
| 运营同学 | 判断选题、修改表达、控制风格、最终发布 | 产出初稿 |
| 销售同学 | 判断客户质量、建立信任、推进成交 | 提炼需求、生成方案、跟进话术 |
| 项目经理 | 关键沟通、资源协调、交付把控 | 整理会议纪要、拆解任务、提醒风险 |
| 创始人 | 找方向、谈大单、做内容 | 承担执行层的很多事情 |

## 四、标准化方法论：70/30 原则

### 4.1 核心观点

**标准化 = 把重复劳动沉淀成 Skill**

| 阶段 | 传统模式 | AI Native 模式 |
|------|----------|----------------|
| 需求 → 初稿 | 技术从零写 | Skill 直接生成 |
| UI → 前端 | 前端逐行写 | Skill 直接转换 |
| 交付 → 案例 | 运营从零整理 | 运营鼠自动生成 |

### 4.2 小程序开发标准化示例

```
过去：
  需求文档 → 技术从零开始 → 搭框架 → 写基础模块 → 配路由 → 写通用逻辑 → 定制化

现在：
  需求文档 → Skill → 项目初稿 → 技术只做 20-30% 定制化调整
  UI 设计图 → Skill → 前端代码 → 技术只做微调
```

### 4.3 标准化的价值

| 价值 | 说明 |
|------|------|
| 降低成本 | 70% 重复劳动由 Skill 承担 |
| 提高效率 | 一个技术同学可以同时管多个项目 |
| 提高利润率 | 收定制化的钱，但大部分成本被 Skill 吃掉 |
| 持续积累 | 每个项目都沉淀可复用的 Skill |

## 五、CoWorker+A2A 项目映射

### 5.1 系统架构映射

| 学习资料概念 | CoWorker+A2A 实现 | 文件位置 |
|-------------|-------------------|----------|
| 谈单鼠 | LLM Router + Agent Routing | `src/lib/agents/llm-router.ts` |
| 项目鼠 | Task Executor + Multi-Agent | `src/lib/agents/task-executor.ts` |
| 运营鼠 | Deliverable System | `src/lib/agents/result-summarizer.ts` |
| 总控 | Orchestrator + Harmony System | `src/lib/harmony/` |
| CRM | Conversation + Message 系统 | `src/app/api/conversations/` |
| PMS | HarmonyTask + TaskRun 系统 | `src/lib/harmony/repository.ts` |
| Skill | Agent Registry + Prompts | `src/lib/agents/registry.ts` |
| 知识库 | Knowledge + Memory 系统 | `src/lib/memory/` |
| 人工审批 | Confirmation Artifact 系统 | `src/lib/harmony/` |

### 5.2 核心流程映射

```
用户需求
  ↓
[Chat API] ──→ 接收需求、保留上下文
  ↓
[LLM Router] ──→ 路由决策（谈单鼠角色）
  ↓
├─ chat_only ──→ LLM 直接回复
├─ delegate_to_agent ──→ 单 Agent 执行
└─ decompose ──→ CEO 分解子任务
  ↓
[Task Executor] ──→ 多轮执行（项目鼠角色）
  ↓
[Deliverable System] ──→ 交付物产出（运营鼠角色）
  ↓
[Confirmation Artifact] ──→ 人工审批门控
  ↓
[Memory + Knowledge] ──→ 经验沉淀
```

### 5.3 安全边界映射

| 学习资料原则 | CoWorker+A2A 实现 |
|-------------|-------------------|
| AI 负责执行，人负责判断 | `requiresHumanConfirmation` 门控 |
| 有所为，有所不为 | Tool Permission Boundary |
| 设止损线 | Task Status 状态机 |
| 标准化沉淀 | Deliverable + Memory 系统 |

## 六、创业方法论

### 6.1 核心原则

| 原则 | 说明 |
|------|------|
| **有所为，有所不为** | 不是每个客户都接，不是每个需求都答应 |
| **设止损线** | 3个月没签客户考虑方向，6个月没营收必须调整 |
| **先框架后填充** | 先搭成熟框架，再让 AI 填充内容 |
| **能力沉淀** | 每个项目都沉淀可复用的 Skill |
| **人从执行者变成管理者** | AI 承担执行层，人做判断和决策 |

### 6.2 踩坑经验

| 坑 | 教训 |
|----|------|
| 什么项目都接 | 能力强的人创业，最大的陷阱是太容易被需要，然后在「被需求」里消耗掉 |
| 不标化 | 不标化，你就是外包公司，利润率上不去 |
| 想过早脱身 | 前三年，创始人的角色不是「抽身」，而是「增长」 |
| 不设止损线 | 创业最大的成本不是钱，是时间 |

### 6.3 成功要素

```
成功 = 清晰的方向 + 一套能用的 AI 体系 + 敢把所有筹码推上桌的勇气

不需要：
  ✗ 大厂背景
  ✗ 百万融资
  ✗ 豪华团队

需要：
  ✓ 一个清晰的方向
  ✓ 一套能用的 AI 体系
  ✓ 一种敢把所有筹码推上桌的勇气
```

## 七、总结

### 7.1 AI Native 的本质

**AI Native 不是让我们看起来更酷，而是让小团队有机会用更少的人，扛起更大的业务。**

### 7.2 核心收获

1. **AI Native 不是用 AI 工具，而是把 AI 训练成岗位员工**
2. **Skill 是 AI 员工的核心，相当于岗位培训手册**
3. **标准化 = 把重复劳动沉淀成 Skill**
4. **人从执行者变成管理者，AI 承担执行层**
5. **系统驱动 > 牛人驱动，公司才能持续运转**
6. **先框架后填充，避免 AI 幻觉导致技术债**

### 7.3 与 CoWorker+A2A 的关系

CoWorker+A2A 项目正是这套 AI Native 理念的技术实现：
- **Agent Registry** = Skill 体系
- **Task Executor** = 项目鼠
- **LLM Router** = 谈单鼠
- **Deliverable System** = 运营鼠
- **Confirmation Artifact** = 人工审批门控
- **Memory + Knowledge** = 知识沉淀
